MLOpsvsLLMOps

The same support-ticket classifier, built two ways — then taken apart stage by stage. One is a model you train. The other is a prompt you steer. Watch what stays the same and what quietly flips.

01Data

What does a “training example” even look like — and does the LLMOps version need one at all?

MLOps data IS the product

  • Collect history
    Pull thousands of past tickets — raw text.
  • Label by hand
    Annotators tag each: Account=0 · Billing=1 · Technical=2. Hundreds–thousands per category.
  • Vectorize
    TF-IDF / embeddings turn text into numbers the model can split.
  • An example =
    a (text, label) pair, versioned in a data store.

LLMOps prompt IS the data

  • Write the schema in words
    The categories + rules live in the prompt, not in a labeled file.
  • Zero labels to ship
    Day-one classifier with no training set at all.
  • Labels return later
    A small labeled set is still useful — but only to measure quality (eval), never to train.
# the prompt is the versioned artifact Categories: Billing → payments, refunds, charges Account → login, profile, password Technical→ bugs, crashes, errors Return JSON: {category, confidence, reason}
Biggest change

Labeled data moves from a prerequisite to an optional eval asset. Add a “Refunds” category? In MLOps you re-label and retrain. In LLMOps you edit one line of the prompt — which is now a code-like artifact that must be versioned, reviewed, and deployed.

02Training / Tuning

There’s a real training loop on the left. What replaces it on the right?

MLOps gradient steps

  • Embed + split
    Vectors in, train/val/test split out.
  • Fit the model
    Predict → compare to ground-truth → compute loss → minimize. Knobs: regularization, iterations, learning rate.
  • “Done” =
    val loss stops improving (early stopping) or target hit, e.g. F1 > 0.90.
  • Cost
    Logistic reg: seconds–minutes. Fine-tuned BERT: hours on a GPU.

LLMOps prompt iteration

  • No training step
    The “loop” is: write prompt → test on sample tickets → read outputs → rewrite → repeat.
  • Tune inputs, not weights
    You change what the model reads at inference: instructions, examples, temperature · top_p · max_tokens.
  • Fine-tune only if needed
    LoRA/QLoRA on dozens–hundreds of examples — when prompt-only accuracy, format consistency, or cost/latency demand it.
  • “Done” =
    prompt edits stop improving eval score. Same idea as early stopping, applied to prompt versions.
Biggest change

The thing you tune flips from model weights to the prompt + inference parameters. A “training run” becomes a “prompt version,” and the iteration cycle drops from GPU-hours to minutes.

03Evaluation

Why isn’t accuracy alone enough once the model also writes a free-text reason?

MLOps one number, mostly

  • Held-out set
    Labeled data never seen in training.
  • Metrics
    Accuracy · precision · recall · F1 · confusion matrix.
  • Ship gate
    F1 > 0.90 and no catastrophic confusions.
  • Re-eval
    after every retrain.

LLMOps multi-dimensional

  • Label accuracy still counts
    Is the category right? That part survives.
  • Format compliance
    Valid JSON? All fields present?
  • Faithfulness
    Is the reason grounded in the ticket, or hallucinated?
  • Confidence ≠ probability
    "confidence": 0.95 is a generated token, not a calibrated number — it can be confidently wrong.
  • Non-determinism
    Same ticket, different call, possibly different answer.
Definition of done

“Good” stops being a single F1 number. It becomes correctness + format + faithfulness + calibration — backed by format checks, LLM-as-judge scoring, and human spot-checks. Accuracy is now one signal among several.

04Deployment

What is the artifact you actually ship — and what does “roll back” mean when the model lives behind someone else’s API?

MLOps ship a binary

  • Package
    Export to ONNX / TorchScript / TensorRT.
  • Serve
    Load into FastAPI · TorchServe · Triton, expose a REST endpoint.
  • Roll back
    Swap the model-registry pointer to the previous version & redeploy.

LLMOps ship a bundle

  • Many artifacts
    prompt · model-id · output schema · guardrails · retrieval config · eval suite — all versioned.
  • Roll back = swap a file
    Point classifier_v7.txt back to v6.txt. No rebuild, no retrain.
  • Model bumps are deploys too
    Vendor moves sonnet-4-5 → 4-6? Re-run the eval suite before promoting — same prompt can behave differently.
prompts/ticket_classifier/ v6.txt # live v7.txt # staged, failing faithfulness model: claude-…-4-6 schema: json_v4
New risks

New artifact types (prompt, schema, guardrails) bring new failure modes: per-token cost that spikes silently, network latency you don’t control, and non-determinism that breaks naive regression diffs.

05Monitoring

In MLOps a broken model usually throws an error. What do you watch when it fails silently instead?

MLOps watch for drift

Signals you track in production:
data drift (PSI/KL) prediction mix shift accuracy · F1 latency p50/95/99 error rate throughput
  • Drift looks like…
    new product slang the model never saw; “Technical” jumps 30%→60% for no reason.
  • Trigger
    retrain when F1 < 0.85 on a weekly labeled sample, or drift score crosses a threshold.

LLMOps watch quality + the vendor

Signals that replace/supplement accuracy:
format-compliance % LLM-as-judge faithfulness token cost/req latency p95 user 👍/👎 · escalations guardrail trips
  • LLM drift — the hard one
    You don’t own the model; the vendor can update it silently. Run a golden set daily; alert if outputs change. That’s your canary.
  • Diffs stop working
    Exact-match breaks on free text → use embedding similarity, entailment, agreement rate / output entropy as variance signals.
Biggest change

“LLM drift.” In MLOps drift comes from your data moving. In LLMOps it can also come from a model you don’t control moving underneath you — caught only by a fixed golden set, not by an error log.

The mental-model shift, in one breath

Going from MLOps to LLMOps moves the center of gravity from weights to prompts: instead of collecting and labeling data to train a model, you ship a prompt on day one and treat it as a versioned, reviewable artifact. Labeled data doesn’t vanish — it relocates from training fuel to an eval set. The hard part shifts downstream to evaluation and monitoring, because the model can be confidently wrong, hallucinated, or off-format with no error thrown, and the same input can vary call to call. And since you no longer own the model, a vendor update can cause silent drift you only catch with a golden set. Net: far less time training, far more spent on prompt iteration, output validation, and watching for failures that never raise an exception.
