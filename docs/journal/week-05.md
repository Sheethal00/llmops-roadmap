# Week 05: Phase 3 - Fine-tuning, Adapters & Model Customization

## Objective

This week focused on Phase 3 of the LLM Ops roadmap: Fine-tuning, Adapters & Model Customization. The goal was to fine-tune a small open-weight model using LoRA on a narrow task, specifically customer support ticket classification.

## Work Accomplished

1.  **Data Preparation (`prepare_dataset.py`):**
    *   Implemented a script to load raw synthetic ticket data (`tickets.jsonl` from `data/raw/`).
    *   Performed an 80/20 train/validation split.
    *   Mapped string labels (e.g., "billing") to integer IDs.
    *   Tokenized the dataset using `meta-llama/Llama-3.2-1B`'s tokenizer, handling truncation and padding.
    *   Saved the processed dataset to `data/processed/` in Hugging Face `datasets` format.
2.  **LoRA Fine-tuning (`train_lora.ipynb`):**
    *   Developed a Google Colab notebook for the actual LoRA/QLoRA training.
    *   Utilized `meta-llama/Llama-3.2-1B` as the base model and applied 4-bit quantization (QLoRA) for parameter-efficient fine-tuning on a T4 GPU.
    *   Configured LoRA with `r=16`, `lora_alpha=32`, and `q_proj`, `v_proj` as target modules.
    *   Trained the model for 2 epochs, tracking metrics (loss, accuracy, F1-macro) with Weights & Biases (W&B).
    *   Pushed the fine-tuned LoRA adapter and tokenizer to Hugging Face Hub (`sheethal00/ticket-classifier-lora-prompt-v2`).
3.  **Model Card (`model_card.md`):**
    *   Created a comprehensive model card detailing the model description, intended use, training data (including generation process and limitations), training details, evaluation results, and limitations.

## Key Decisions and Tradeoffs

*   **Model Selection:** `meta-llama/Llama-3.2-1B` was chosen due to its small size, making it suitable for fine-tuning on resource-constrained environments like Google Colab's free tier and for eventual CPU-only inference (as per project hardware constraints).
*   **Fine-tuning Strategy:** QLoRA was selected over full fine-tuning to minimize computational cost and memory footprint during training, leveraging the parameter-efficient approach.
*   **Hardware Split:** Data preparation was performed locally to allow for iterative development and larger datasets if needed, while the compute-intensive QLoRA training was offloaded to Google Colab's T4 GPU, respecting the CPU-only constraint of the local machine for training.
*   **Synthetic Data:** The dataset was synthetically generated by `claude-sonnet-4-6`. A key learning was the importance of clear, positive definitions for all classification categories, especially the 'general' category, to avoid ambiguous data generation and improve model performance.
*   **Evaluation Focus:** Evaluation focused on accuracy and F1-macro. Initial analysis of the confusion matrix revealed minor confusion between 'technical' and 'account' categories, suggesting potential areas for data refinement or prompt engineering in `generate_dataset.py` if further improvements are desired.
*   **Experiment Tracking:** Weights & Biases (W&B) was integrated into the Colab notebook for robust experiment tracking, capturing hyperparameters, loss curves, and evaluation metrics.
